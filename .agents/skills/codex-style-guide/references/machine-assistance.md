# Writing for machine-assisted experiences

Source: <https://doc.wikimedia.org/codex/latest/style-guide/machine-assistance.html>

How and when to signal machine-learning-assisted content in the UI,
aligned with Wikimedia's value of human-created knowledge.

## Content principles

- **Center the human, not the technology.** Focus on what the person is
  doing. Active voice for people (*add, write, edit, review*); passive
  for the machine (*generated by*). Don't frame the system as a partner
  or teammate.
- **Give just enough information.** Prioritise what helps the task; keep
  it scannable; push technical detail behind a descriptive *Learn more*
  link (e.g. a model card).
- **Be transparent.** Be specific about what the system does; show
  metadata that helps (e.g. confidence per prediction) and an
  information trail (sources, keywords, inputs). Avoid vague claims of
  "intelligence"; prefer specific terms ("small language model",
  "classifier model") over a blanket "AI".

## Choose a level of disclosure

Risk comes from the **data**, the **model**, *and* how output is
**presented** — not the model alone. Disclose more when output is less
predictable/interpretable and the application is complex or higher-risk.

- **Detailed disclosure** — for generated content that could be mistaken
  for human-made or where errors erode trust. Include early disclosure,
  a label (when appropriate), a disclaimer, clear instructions, a
  feedback mechanism, and supporting detail (sources/keywords/reasoning).
  *E.g. long-form generated text, machine summaries, high-stakes
  translations, abuse-prevention actions, personalized recommendations.*
- **Basic disclosure** — for predictable, interpretable, lower-risk
  output. *E.g. machine translation, editing suggestions, non-personalized
  recommendations like trending/similar articles.*

## Components for disclosure (by mode: Flag / Recommend / Generate)

- **Titles** — sentence case, action-oriented, named for the task not
  the model. Action form: `[verb] [noun]` ("Add a link", "Check a
  fact"); Generate: `[verb] machine [noun]` ("Review machine
  translation"). Feature/tool name form: "Reference check", "Suggested
  articles", "Machine translation".
- **Labels & icon** — optional; prefer a text label over the `robot`
  icon. Skip labels for spellcheck-like Flag/Recommend cases. For
  **Generate**, be explicit: "Identified by a [description] model"
  (link the model card).
- **Disclaimers** — needed for **Generate** always; for **Recommend**
  in editor instructions; for **Flag** only when it triggers a mandatory
  change/automated action. Pattern: *"Machine [output] can be wrong or
  incomplete. [How it can be wrong.] [What the person can do.]"*
  E.g. "Machine translations can be wrong or incomplete. Check important
  information."
- **Feedback** — important for Generate and crucial for automated
  actions. Use a close/discard "X", a short yes/no, a dropdown, or a
  review form. Explain how feedback is used. **Avoid thumbs up/down.**
- **Buttons** — sentence case, verbs, one or two words, matching the
  card/dialog title ("Add image", "Generate list", "View details").
  Don't default to a generic "Submit".
- **Links** — descriptive out of context; "Learn more about machine
  suggestions" rather than a bare "Learn more".

## Checklist

- Is the focus on the user's task?
- Is the explanation proportional to the risk?
- Is the language plain, neutral, and translatable?
- Is assistance disclosed at the right level?
- Are generated outputs labelled clearly when needed?
- Is there a feedback path when the use case needs one?
